    Support for generating tabular HRD data
    
    The 'Generator::tabular' method instructs the HRD generator to format
    the contained nodes in tabular way. Two forms are supported: In the
    presence of nested nodes, the nodes appear as table columns and
    attributes appear densely. With only one node level, attributes are
    formatted as a table. Trailing optional attributes are displayed
    densely.
    
    Fixes #5708

Simplified configuration of POSIX environment and arguments
===========================================================

; @nfeske

commit 12f5b4c8db280b4ed6912e85f19e17a0c8ed7152
Author: Norman Feske <norman.feske@genode-labs.com>
Date:   Sat Oct 25 11:25:26 2025 +0200

    posix: support optional arg 'name' attribute
    
    Since tag-value argument pairs are technically two separate arguments,
    they have to appear as separate arg nodes in the configuration, e.g.:
    
    + config
      + arg | : -geometry
      + arg | : 1024x768
    
    This wastes vertical space and does not convey well that both lines
    belong together.
    
    By combining an optional 'name' attribute with quoted content, however,
    tag-value arguments can nicely be written on a single line:
    
    + config
      + arg -geometry | : 1024x768
    
    Fixes #5726

Author: Sebastian Sumpf <sebastian.sumpf@genode-labs.com>
Date:   Tue Nov 18 11:45:39 2025 +0100

    libdrm: add vtable to Gpu::Vram
    
    Commit "libdrm: move Gpu::Vram into separate namespace"
    (db02e04d0c5465579a3d7d66dac17f2fc5ca830a) lead to an implementation
    where both, lima and etnaviv, inherit from an empty Gpu::Vram class.
    This causes the problem that the ID-space in both implementations is of
    type Gpu::Vram, leading to the respective destructors (Lima::~Vram,
    Etnaviv::~Vram) not being called upon destruction, and thus, not freeing
    GPU memory.
    
    Featured in this commit:
    
    * move the Gpu::Vram definition into separate header which is used by
      entaviv/lima
    
    * add virtual destructor to Gpu::Vram
    
    issue #5748
